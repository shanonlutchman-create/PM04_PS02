📋 Part 1: Process Analysis and Design

Task 1 — Decide Bot Steps   (PA0201)

✓ Marks Available: 5

The software robot will execute the following sequential steps to automate the MzansiConnect customer onboarding process:

Step 1 — Initialisation & Open Applications

  1. Log the process start timestamp: "MzansiConnect Customer Onboarding process started at yyyy-MM-dd HH:mm:ss"
  2. Read the customer data from customer_data_south_africa.csv into DataTable dt_CustomerData
  3. Log confirmation that the CSV was loaded successfully, including the total number of records found
  4. Build an empty Exceptions DataTable (dt_Exceptions) with columns: CustomerID, Reason, Timestamp
  5. Open the CRM web portal in a browser using the Open Browser activity

Step 2 — Log In to the CRM Portal

  1. Navigate to the CRM login URL
  2. Type the CRM username into the username field (sourced from a secure UiPath Orchestrator Asset — never hardcoded)
  3. Type the CRM password into the password field (sourced from a secure credential store)
  4. Click the "Login" button
  5. Verify that the CRM dashboard has loaded successfully before proceeding (post-login verification)

Step 3 — Iterate Through Each Customer Record

  1. Begin a For Each Row loop through every row in dt_CustomerData
  2. For each row, extract and assign: CustomerID, Name, Surname, SA_ID_Number, Phone_Number (cleaned to digits only using Regex)
  3. Validate the record:
    • If SA_ID_Number is blank → log warning, add to Exceptions, skip to next row
    • If phone number (digits only) ≠10 digits → log warning, add to Exceptions, skip to next row
  4. Log the current CustomerID being processed (with PII fields masked)

Step 4 — Enter Data into the CRM

  1. Click the "Add New Customer" button on the CRM portal
  2. Type strName into the Name field
  3. Type strSurname into the Surname field
  4. Type strSA_ID into the SA ID Number field
  5. Type strPhone into the Phone Number field
  6. Click the "Submit" button
  7. Wait for and verify the confirmation message appears on screen
  8. Take a screenshot of the confirmation message and save it to Data\Screenshots\[CustomerID]_confirmation.png
  9. Log successful submission and increment intSuccessCount

Step 5 — Logging Out & Closing Applications

  1. After all rows are processed, click Logout on the CRM portal
  2. Close the browser
  3. Write dt_Exceptions to Data\Exceptions\Skipped_Records.csv
  4. Log the final summary: Total records, Success count, Skipped count
  5. Log the process end timestamp

B. PII Identification and Handling

Which fields are PII?
The following fields are classified as Personal Identifiable Information (PII) under South Africa's POPIA (Protection of Personal Information Act):

FieldWhy It Is PII
SA_ID_NumberUniquely identifies a South African citizen — highly sensitive government-issued number
Phone_NumberDirectly links to and identifies an individual
Name + SurnameDirectly identifies the customer by name
EmailPersonally identifies the customer and enables direct contact

How will PII be prevented from being logged?

Three complementary strategies are applied:

  1. Masking in Log Messages — Only partial data is shown in logs:
    • SA ID: ********* + last 4 digits  (e.g. *********3087)
    • Phone: ****** + last 4 digits  (e.g. ******4567)
    • Full names are never written to logs directly
  2. UiPath Project-Level Log Exclusions — The excludedLoggedData property in project.json is configured to exclude "Private:*" and "*password*", preventing accidental capture of sensitive variable values at the framework level.
  3. Exception messages reference only CustomerID — a non-sensitive reference key. The actual PII value is never written to the exception log or output report.

Task 3 — Exception Handling   (PA0203 & IAC0203)

✓ Marks Available: 6

Scenario A — Missing Data: SA_ID_Number is blank

Exception Type: Business Exception

A Business Exception occurs when the data itself does not meet the required criteria for processing. A missing SA ID Number is a data quality issue — the process cannot complete for this record, but the overall automation can and should continue.

Handling Plan:

  1. Before attempting to submit data to the CRM, the bot checks: If String.IsNullOrWhiteSpace(strSA_ID)
  2. If True (missing):
    • Log a Warning: "Row [CustomerID] skipped: SA_ID_Number is missing"
    • Add a row to dt_Exceptions: {CustomerID, "Missing SA_ID_Number", Timestamp}
    • Increment intSkipCount by 1
    • Use a Continue activity to skip to the next row — no CRM form is opened

Verification before continuing: The bot must confirm strSA_ID is non-empty and non-whitespace before proceeding to any CRM data entry steps.

Scenario B — System Error: CRM Portal Unavailable (HTTP 503)

Exception Type: System Exception

A System Exception is caused by an external technical failure outside the bot's control. The CRM server being temporarily unavailable is not a data issue — it is an infrastructure failure that blocks all further processing.

Handling Plan:

  1. Wrap the CRM browser interaction steps inside a Retry Scope activity configured to retry up to 3 times with a 30-second delay between each attempt.
  2. If the portal remains unavailable after all retries:
    • Log an Error: "CRM portal unavailable after 3 retry attempts. Process is being terminated."
    • Send an email alert or raise a UiPath Orchestrator alert to notify the supervisor/CSR
    • Write any already-collected exception records to the CSV before stopping
    • Terminate the process — do not attempt to process further records as none can be submitted

Verification before continuing: Before entering any customer data, the bot must verify the CRM dashboard has fully loaded (e.g. confirm the "Add New Customer" button is visible and enabled on screen).

Scenario C — Incorrect Format: Non-numeric Phone Number (e.g. 083-456-7890)

Exception Type: Business Exception

This is a data formatting issue. The data exists but is in an incorrect format. This is a Business Exception because the source data does not conform to the expected standard — however, it may be recoverable through data cleansing.

Handling Plan:

  1. When extracting the phone number, immediately apply a Regex cleansing step to strip all non-numeric characters:
    strPhone = Regex.Replace(CurrentRow("Phone_Number").ToString.Trim, "[^0-9]", "")
    This converts 083-456-7890 → 0834567890.
  2. After cleansing, validate the length: If strPhone.Length <> 10
  3. If still invalid after cleansing:
    • Log a Warning: "Row [CustomerID] skipped: Phone invalid after cleansing (length: X)"
    • Add row to dt_Exceptions: {CustomerID, "Invalid Phone Number", Timestamp}
    • Increment intSkipCount and use Continue to move to the next record
  4. If valid after cleansing (length = 10): proceed normally using the cleaned value for CRM submission.

Verification before continuing: The bot must confirm strPhone.Length = 10 (after Regex cleansing) before entering the phone number into the CRM form.

Task 4 — Reusability   (IAC0201)

✓ Marks Available: 4
FieldDetail
Component NameValidateAndSubmitCustomer.xaml
DescriptionA reusable workflow component that accepts a single customer's data as input arguments, performs all business validation checks (SA ID present, SA ID Luhn checksum valid, phone number format valid), and — if the record passes validation — submits the customer's details to the CRM web portal and captures a confirmation screenshot. Returns a success/failure status and reason message, making it callable from any parent automation that needs to onboard a customer into the MzansiConnect CRM.

Input Arguments

Argument NameDirectionTypeDescription

Output Arguments

Argument NameDirectionTypeDescription
out_IsSuccessOutBooleanTrue if record was submitted; False if skipped/failed
out_SkipReasonOutStringPopulated with the reason if out_IsSuccess = False; empty string if successful

Pre-Conditions

  • The CRM web portal must be open and the user already logged in before this component is called
  • The parent workflow must supply non-null values for in_CustomerID, in_Name, and in_Surname

Post-Conditions

  • If out_IsSuccess = True: a new customer record exists in the CRM and a screenshot has been saved to in_ScreenshotFolder
  • If out_IsSuccess = False: no data was entered into the CRM; out_SkipReason contains the specific validation failure message for the parent workflow to log

Task 5 — Development   (AK0201–AK0205)

AK0201 — Scripting / Programming Language

The workflow is developed in UiPath Studio using VB.NET as the expression language. Custom code logic is embedded via the Invoke Code activity for SA ID Luhn checksum validation — a loop-based algorithm that cannot be expressed as a single-line VB.NET expression. Regex is also used inline (System.Text.RegularExpressions.Regex.Replace) to clean phone numbers of non-numeric characters directly within Assign activity expressions.

AK0203 — Processes and Workflows

The solution follows the PDD closely. Main.xaml is structured as a single linear sequence that first initialises data, then loops through records with a per-row TryCatch, and finally outputs the exceptions report. The design separates data extraction, validation, CRM submission, and reporting into clearly named, logically ordered activities — making the workflow readable and maintainable.
